University of Westminster — Client Server Architecture

Smart Campus Sensor API

A fully functional RESTful backend system that simulates a smart university campus, managing rooms, IoT sensors, and live sensor readings using a scalable Java-based architecture.

Project Overview

A backend REST API built using Java, JAX-RS, Jersey, and Grizzly that simulates a smart campus system. It manages rooms, sensors, and real-time sensor readings for a university environment.

System Concept

The system models a real-world IoT infrastructure:

Rooms → Lecture halls, labs, study spaces
Sensors → Temperature, CO2, occupancy devices
Sensor Readings → Live data collected over time

REST API Design

/api/v1/rooms → Manage rooms
/api/v1/sensors → Manage sensors
/api/v1/sensors/{id}/readings → Sensor data history

System Architecture Flow

Client (Postman / Browser)
↓
JAX-RS REST Layer (Jersey)
↓
Service Layer (Business Logic)
↓
In-Memory Data Store (ConcurrentHashMap)

Architecture Highlights

✔ Embedded Grizzly HTTP Server
✔ Jersey JAX-RS REST framework
✔ Maven Shade build (single JAR)
✔ ConcurrentHashMap in-memory datastore

Advanced Design Patterns

✔ Sub-resource locators for nested sensor readings
✔ HATEOAS navigation support
✔ Centralized exception handling
✔ Request/response logging filter

Error Handling System

409 → Conflict (sensor blocking room deletion)
422 → Invalid request payload
403 → Forbidden operation
500 → Safe global error handler (no stack traces exposed)

Design Decisions

✔ Why JAX-RS?
Lightweight and annotation-driven REST framework.

✔ Why Grizzly?
Embedded server for quick deployment without external setup.

✔ Why ConcurrentHashMap?
Ensures thread-safe operations for concurrent requests.

Key Technical Strengths

✔ Thread-safe concurrent data handling
✔ RESTful best practices (status codes, idempotency)
✔ Secure API design principles
✔ Clean separation of concerns
